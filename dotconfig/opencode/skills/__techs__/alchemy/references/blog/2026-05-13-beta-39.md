# alchemy@2.0.0-beta.39 (2026-05-13)

Fix-only release covering three issues people hit after beta.38.

## `Cloudflare.Vite` inlines `VITE_*` env into the bundle

`env` was forwarded only as runtime Worker bindings — values never made it into the client/SSR bundle. SPAs reading `import.meta.env.VITE_API_URL` fell back to hardcoded defaults.

The build step now passes `props.env` through Vite's `define` hook:

```ts
const web = yield* Cloudflare.Vite("Web", {
  env: { VITE_API_URL: worker.url.as<string>() },
});

// SPA: `import.meta.env.VITE_API_URL` now contains the deployed Worker URL.
```

Rules match Vite's own:
- Only `VITE_`-prefixed keys are inlined as `import.meta.env.<key>`
- Non-`VITE_` entries stay out of the bundle but remain as runtime Worker bindings
- `Redacted` values are unwrapped only when `VITE_`-prefixed (explicit opt-in to public bundle)

## Cloudflare Worker HTTP adapter — Effect's handled lifecycle

The Worker HTTP adapter previously hand-rolled the `HttpServerRequest` / `HttpServerResponse` bridge, diverging from Effect's `HttpEffect.toHandled` / `toWebHandler`. Most visibly, `RpcServer.toHttpEffect` hung when served through Alchemy.

The adapter now runs handlers through Effect's standard HTTP lifecycle while preserving Worker-specific behavior:

- `cf-connecting-ip` → `remoteAddress`
- Original Cloudflare `Request` service
- Patched `request.raw` access
- Existing 500 response mapping

Plus stream-scope transfer, HEAD body suppression, and request-abort interruption.

`RpcServer.toHttpEffect` is unblocked.

## `SendEmail` binding type inference

beta.38 added `Cloudflare.SendEmail` but two wiring sites in `Worker.ts` were missed — binding type inference didn't know about it, and deploy-time metadata didn't emit a `send_email` entry. Fixed.
