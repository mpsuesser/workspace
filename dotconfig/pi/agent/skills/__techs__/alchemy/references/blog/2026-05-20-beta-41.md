# alchemy@2.0.0-beta.41 (2026-05-20)

Mostly an internal bridge rewrite. The Worker entrypoint is now a thin generated shell that calls into a type-checked `WorkerBridge.ts`. Scope construction is linear, per-request, and no longer mixes promises across the workerd I/O boundary.

## RPC and HTTP API â€” per-request scopes

The previous Worker runtime cached the user's Worker Effect once as a module-scope `Promise`, then mixed scopes between `WorkerBundle`, `WorkerRuntime`, and `HttpServer` per request. Three production failures:

1. **Too many scopes.** Each layer created its own `Scope`; close order between them wasn't well-defined. Finalizers ran against torn-down scopes or never ran.
2. **Promises crossed the workerd I/O boundary.** A single cached promise was awaited from multiple requests â€” workerd aborts those as "wrong I/O context" (hung `fetch`es, dropped responses).
3. **Global vs request-level scope confusion.** Construction-time and per-request layers merged in the wrong order; request-level services occasionally resolved from the global scope with stale state.

beta.41 refactors the entrypoint into [`WorkerBridge.ts`](https://github.com/alchemy-run/alchemy-effect/blob/main/packages/alchemy/src/Cloudflare/Workers/WorkerBridge.ts). Scoping is linear: one `Scope.makeUnsafe()` per request, provided into the user's Effect, closed via `ctx.waitUntil` when the response finishes. Nothing cached as a promise; the user's Worker Effect re-evaluates per request against its own scope.

> Trades a small amount of per-request setup for correctness. The previous "cache once" path was the source of the bugs.

The same fix lands across `Cloudflare.Worker`, `Cloudflare.DurableObjectNamespace`, and `Cloudflare.Workflow`, so the schema-driven pattern from the [HTTP API](../guides/effect-http-api.md) and [RPC](../guides/effect-rpc.md) guides now holds together under real traffic.

## The schema-driven pattern, end-to-end

One schema feeds both HTTP API endpoints and RPC procedures. Worker exposes either transport. DO speaks the same protocol back via `Cloudflare.toHttpClient(stub)`.

```ts
// Bind a DO and get a typed fetcher per instance
const tasksDO = yield* TasksObject;

const getDOClient = (id: string = "default") =>
  HttpApiClient.makeWith(TaskDOApi, {
    baseUrl: "http://localhost",
    httpClient: Cloudflare.toHttpClient(tasksDO.getByName(id)),
  });
```

`baseUrl` is a placeholder â€” `toHttpClient` short-circuits to `tasksDO.getByName(id).fetch`. The same wrapper plugs into `RpcClient.layerProtocolHttp` for RPC.

## Other fixes

- **`WorkerProps.env` flows into `InferEnv`** â€” typing `worker.env.MY_VAR` picks up keys declared in `env: { ... }`, not just bindings
- **`WorkerExecutionContext` + `ExecutionContext` in fetch types** â€” `ctx.waitUntil(...)` reachable from the handler without a cast
- **DO methods receive the Worker env** â€” RPC methods on `DurableObjectNamespace` were running with empty `env` context
- **`Redacted` survives sidecar RPC serialization** â€” sidecar was unwrapping `Redacted` during JSON round-trips, leaking secrets in dev logs
- **`lightningcss` + `fsevents` externalized** in the Worker bundler (platform-specific native deps)
- **Worker teardown force-deletes scripts** with stuck Durable Object migrations
- **Eager terminal status events** â€” CLI shows `pending` for blocked resources and emits terminal status as soon as the reconcile finishes
